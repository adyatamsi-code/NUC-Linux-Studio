# Project Rules — NUC Linux Studio

## Rule 1: No Secure Boot
This system runs **Fedora 44 with Secure Boot DISABLED**. The user installed Fedora without Secure Boot. Do NOT troubleshoot or suggest Secure Boot/MOK enrollment steps. Module signing warnings ("tainting kernel") are harmless and expected.

## Rule 2: No qc71_laptop or asus_wmi
The `qc71_laptop` and `asus_wmi` modules are **blacklisted** via `/etc/modprobe.d/nuc-studio.conf`. Our `nuc_wmi` driver replaces qc71_laptop entirely and handles the same WMI GUIDs. Loading both causes duplicate event handling and input device conflicts.

## Rule 3: keyd Grabs All Input Devices
keyd v2.6.0 is installed and **EVIOCGRAB-grabs ALL input devices** regardless of its `[ids]` config. There is NO way to exclude the NUC WMI input device from keyd's grab. Consequences:
- Daemons CANNOT use evdev to read from "NUC WMI input device" — keyd consumes all events
- `/dev/kmsg` `select()`/`poll()` is broken on kernel 6.19.x — does not wake on new messages
- **Solution**: Use `dmesg --follow` (subprocess pipe) to detect WMI events like "keyboard backlight changed"
- Do NOT monitor "keyd virtual keyboard" from daemons — it fires for every keystroke (25%+ CPU)

## Rule 4: Fan RPM Limits
- **CPU fan max**: 5780 RPM
- **GPU fan max**: 5500 RPM

## Build Environment
- **OS**: Fedora 44 (fc44)
- **Kernel**: 6.19.x series (x86_64)
- **Python**: 3.14
- **Shell**: bash
- **Install path**: `/opt/nuc-linux-studio/`
- **DKMS module**: `nuc_wmi/1.0` (includes `nuc_wmi.ko` + `ite8291r3.ko`)

## Known Fedora 44 Kernel-Devel Bug
The `kernel-devel` package for 6.19.14-300.fc44 ships **without generated arch headers** (`cpufeaturemasks.h`, `orc_hash.h`). Fix: `sudo dnf reinstall kernel-devel-$(uname -r)` — the reinstall populates the missing files. If build fails with "cpufeaturemasks.h not found", reinstall kernel-devel first.

## Rule 5: Always Backup Before Modifying
**Always** backup files before modifying them (especially driver source, daemon code). Place backups in `backup/` with date subfolder. Never overwrite working code without a backup — previous working code was lost this way.

## Rule 6: Use Gemini for Online Research
When stuck on hardware behavior or need firmware/EC documentation, ask the user to query Gemini (which can search online). Don't guess at hardware behavior.

## Touchpad Toggle — Hardware Findings & Command Log

### What WORKS ✅
| Action | Result |
|--------|--------|
| `Fn+F7` toggle (dmesg watcher) | Toggles touchpad on/off, LED reflects state correctly |
| HID write `0x03` (Report ID 7) | Touchpad ON, LED OFF |
| HID write `0x00` (Report ID 7) | Touchpad OFF, LED ON — **but kills digitizer, no double-tap possible** |
| EC `touchpad_enabled` sysfs write `1`/`0` | Sets CTRL_4 bit 6 |
| gsettings `send-events` enabled/disabled | Software-level touchpad enable/disable |
| `dmesg --follow` subprocess pipe | Reliably detects "touchpad toggle pressed" from Fn+F7 |
| Double-tap to DISABLE | Works — firmware sets HID to 0x00, LED turns on, touchpad off |

### What DOES NOT WORK ❌
| Action | Result |
|--------|--------|
| Double-tap to RE-ENABLE | **DEAD** — HID 0x00 kills digitizer hardware, firmware can't detect tap to re-enable |
| HID value `0x01` | LED OFF, touchpad OFF — useless |
| HID value `0x02` | LED ON, touchpad OFF — same as 0x00 without LED benefit |
| Any HID value for LED ON + touchpad alive | **Does not exist** — all tested (0x00-0x03), none provide LED ON with touchpad hardware alive |
| EC CTRL_4 bit 7 (with bit 6=1) | LED ON but cursor doesn't move (bit 6 was already set = touchpad off, inconclusive test) |
| `/dev/kmsg` select()/poll() | Broken on kernel 6.19.x — never wakes |
| `systemctl restart nuc-touchpad-daemon` | Service unit does not exist; daemon runs via `/opt/nuc-linux-studio/` directly |
| Modifying driver source + DKMS rebuild for EC changes | **DANGEROUS** — caused keyboard to stop working, touchpad to break, backspace issues. Restored from backup. |

### NOT YET TESTED ⏳
| Action | Notes |
|--------|-------|
| EC CTRL_4 bit 7 with bit 6 CLEARED | Previous test had bit 6=1 (touchpad already off). Need to test writing `0x81` (bit7=1, bit6=0) to see if bit 7 controls LED independently while touchpad stays alive |
| HID poller to detect double-tap state change | Polling HID feature report to see firmware toggle without writing HID 0x00 |

### Key Constants
- **HID device**: `/dev/hidraw3` (UNIW0001:00)
- **HID Feature Report ID**: 7 (2 bytes: `[report_id, value]`)
- **EC CTRL_4 address**: `0x07A6` (page 0x07, offset 0xA6)
- **EC CTRL_4 bit 6**: `TOUCHPAD_TOGGLE_OFF`
- **EC CTRL_4 bit 7**: Unknown — possibly LED control (needs clean test)
- **debug_ec sysfs**: `/sys/devices/platform/nuc_wmi/debug_ec` — safe EC read/write without driver rebuild
- **touchpad_enabled sysfs**: `/sys/devices/platform/nuc_wmi/touchpad_enabled`

### The Core Problem
Double-tap can DISABLE (firmware writes HID 0x00 → LED on, touchpad off) but cannot RE-ENABLE because HID 0x00 puts the digitizer into D3 low-power mode, killing all touch detection including the double-tap gesture needed to re-enable. **The old code did work** — need to find what approach was used (possibly HID poller that detects firmware state changes without writing 0x00, or EC-only LED control).

### Commands That Don't Work (Don't Retry)
```
sudo systemctl restart nuc-touchpad-daemon   # unit does not exist
sudo systemctl status nuc-touchpad-daemon     # unit does not exist
```

### Commands That Work
```
ps aux | grep touchpad_daemon | grep -v grep                    # find running daemon
sudo pkill -f touchpad_daemon.py                                 # kill daemon
sudo /usr/bin/python3 -u /opt/nuc-linux-studio/backend/touchpad_daemon.py  # start daemon
sudo cp backend/touchpad_daemon.py /opt/nuc-linux-studio/backend/touchpad_daemon.py  # deploy
cat /tmp/touchpad_daemon.log                                     # check daemon log (if redirected)
sudo dmesg | grep -i touchpad                                    # check kernel touchpad messages
cat /sys/devices/platform/nuc_wmi/touchpad_enabled               # read EC touchpad state
echo "a6" | sudo tee /sys/devices/platform/nuc_wmi/debug_ec      # read EC CTRL_4
echo "a6 XX" | sudo tee /sys/devices/platform/nuc_wmi/debug_ec   # write EC CTRL_4
```
